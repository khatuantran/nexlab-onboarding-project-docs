# CR-004 — Image storage: Fly volume → Cloudinary

## Summary

Migrate user-uploaded screenshot storage from Fly persistent volume (`/data/uploads`) to **Cloudinary** free tier (25 GB storage + 25 GB BW, no CC). Implemented in two phases — Phase 1 destroys the volume immediately to stop the $0.26/month billable cost; Phase 2 lands the Cloudinary SDK wiring later. Between phases the upload route writes to the container's ephemeral filesystem (uploads survive only until the next machine restart), accepted as a known transient gap.

## Motivation

- **$0.26/month Fly volume cost** documented in the dashboard preview invoice (2026-05-14) — small, but the only billable line for this pilot.
- **BUG-003 root cause** also touches the `/api/v1/uploads/:id` serve path; routing image bytes through Cloudinary's CDN sidesteps the cross-origin/auth fragility entirely and gives transformations (resize / WebP) for free.
- **Pilot budget = $0** per [CR-003](CR-003.md) constraint. Volume cost is the only line item we can drop without breaking anything load-bearing.
- **Free tier headroom**: pilot scale (≤ 100 features × ≤ 5 images × ~500 KB ≈ 250 MB total) uses < 1% of Cloudinary's 25 GB.

## Proposed change

### Before

- `apps/api/src/routes/uploads.ts` POST writes binary to `${UPLOAD_DIR}/<uuid>.<ext>` (defaults to `/data/uploads/<uuid>.<ext>` in prod).
- `apps/api/src/routes/uploads.ts` GET reads same path via `res.sendFile`.
- `apps/api/fly.toml` mounts `uploads_volume` (1 GB, region SIN) at `/data/uploads`.
- Response body returns `url: /api/v1/uploads/${row.id}` (relative).

### After

**Phase 1 (this CR — ships immediately)**:

- `apps/api/fly.toml` drops the `[[mounts]]` block. Container runs without a persistent volume.
- `fly volumes destroy vol_vly2yydkd99pkxm4` executed out-of-band after the new release rolls out.
- Upload route code unchanged in Phase 1 — `writeFile` lands in the container's ephemeral `/data/uploads` directory (still `mkdir -p`'d in [apps/api/Dockerfile](../../apps/api/Dockerfile)). Files survive until the next machine restart (auto-stop after ~30 s idle, deploy, or crash).
- The 3 existing volume files (2.7 MB total, listed 2026-05-14 in BUG-003 §Reproduction) are lost. Per BUG-003, none of them ever rendered on prod (cross-origin `<img>` cookie path failed before the fix shipped), so production data loss is nil even though the DB rows + markdown links remain.

**Phase 2 (deferred to a follow-up CR commit or new US)**:

- Add `cloudinary` Node SDK dependency to `apps/api`.
- Rewrite `apps/api/src/routes/uploads.ts` POST to stream the multer buffer through `cloudinary.uploader.upload_stream`. Return Cloudinary's `secure_url` (absolute `https://res.cloudinary.com/...`) as `response.url`.
- DELETE `createUploadsReadRouter` entirely — Cloudinary's CDN serves the binary, no proxy needed. Remove its wiring from [apps/api/src/index.ts](../../apps/api/src/index.ts).
- DB migration: add `uploads.cloudinary_public_id` column; backfill nullable.
- `apps/api/.env.example` + Fly secret: `CLOUDINARY_URL=cloudinary://<api_key>:<api_secret>@<cloud_name>`.
- FE: zero change — [apps/web/src/lib/markdown.ts](../../apps/web/src/lib/markdown.ts) already passes `https?:` URLs through the sanitizer untouched, so Cloudinary's absolute URLs render directly.
- Old `/api/v1/uploads/:id` route can stay temporarily as a 404 stub or be removed outright (no callers after the FE point at Cloudinary URLs).

## Alternatives

| Option                                   | Pros                                   | Cons                                                                                          | Lý do không chọn                                                        |
| ---------------------------------------- | -------------------------------------- | --------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| **Keep Fly volume**                      | Zero code change, status quo           | $0.26/mo billable line; BUG-003 root cause (cross-origin BE serve) only partially mitigated   | User explicitly wants to drop the line; CDN wins are real               |
| **Tigris (Fly partner)** 5 GB free       | Same datacenter (SIN), Fly-native auth | No CDN outside SIN; 5 GB < Cloudinary's 25 GB; transformations not free                       | Cloudinary cheaper at scale + better DX                                 |
| **Cloudflare R2** 10 GB free             | 10 GB storage, free egress             | CC required to verify account; no native image transforms; CDN via Workers adds another layer | CC friction; transform features still useful                            |
| **Supabase Storage** 1 GB free           | Simple S3-API                          | 1 GB hard cap; trial-credit limits                                                            | Headroom too small                                                      |
| **Inline base64 in markdown body**       | Zero infra                             | Bloats DB rows; no CDN; breaks search snippet sanitizer                                       | Architecturally wrong                                                   |
| **Migrate to Cloudinary first (no gap)** | No transient ephemeral-fs window       | ~5 h up-front before volume can be destroyed; $0.26 keeps ticking that day                    | User chose speed over no-gap (CR-004 §Decision); 3 ảnh đã chấp nhận mất |

## Decision

**Approved 2026-05-14** — Phase 1 ships in the next 3 commits (spec → fly.toml → progress sync). Phase 2 deferred to a follow-up CR or US opened when user revisits image work. User explicitly accepted (1) permanent loss of the 3 existing volume files, (2) silent ephemeral-fs data loss for any uploads attempted between Phase 1 ship and Phase 2 ship, and (3) sequence A (destroy volume first, migrate later) over the no-gap sequence B.
