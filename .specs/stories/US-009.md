## User perspective

> As an **authenticated user (any role)**, I want **to view + edit my own profile (display name, password, avatar)** trên 1 page riêng, so that **tôi tự branding tài khoản + đổi password không cần phiền admin**.

## Why this story now?

UserMenu (AppHeader v2, `f1dfa40`) hiện có 2 item disabled placeholder ("Hồ sơ của tôi", "Cài đặt") kèm label "Sắp ra mắt". User báo: admin reset password cho từng người = friction; users muốn tự đổi. Avatar gradient initials đẹp nhưng pilot thực có 5+ user trùng initials ("NL" = Ngọc Linh + Nhật Long) gây nhầm. Self-service profile là tile nhỏ scope rõ, ROI cao, không blocker.

## Scope in

- Schema: `users.avatar_url TEXT NULL` (migration `0009`).
- API (`requireAuth` only, không admin gate):
  - `GET /api/v1/me` → `AuthUser` shape extended với `avatarUrl`.
  - `PATCH /api/v1/me` → đổi `displayName` (chỉ field này).
  - `POST /api/v1/me/password` → `{ oldPassword, newPassword }` → 204; purge other Redis sessions, keep current.
  - `POST /api/v1/me/avatar` → multipart `file` → Cloudinary CDN folder `avatars/`; persist URL; return `{ avatarUrl }`.
- FE:
  - Page `/profile` (3 section: Profile · Security · Avatar).
  - `UserMenu` "Hồ sơ của tôi" item enabled → `Link to="/profile"`.
  - `Avatar` component extend với `imageUrl` prop, fallback initials.
  - Header avatar refresh real-time sau mutation (invalidate `authKeys.me`).

## Scope out

- **Email change** — defer; cần verification flow phức tạp. Admin sửa qua DB nếu cần.
- **Role self-change** — explicit security boundary.
- **MFA / 2FA** — v2.
- **Avatar crop UI** — Cloudinary auto-resize transform query đủ.
- **Old-avatar orphan cleanup** trên Cloudinary — accept drift v1.
- **"Cài đặt" menu item** — placeholder giữ disabled.
- **Profile picture trong AvatarStack + ProjectAvatar** — chỉ render trong UserMenu + dedicated Avatar instance v1.
- **Audit log** cho profile edit / password change.

## Acceptance criteria

### AC-1 — UserMenu navigate to /profile

**Given** user đang login
**When** click avatar header → click "Hồ sơ của tôi" item
**Then** browser navigate `/profile`
**And** menu item KHÔNG còn disabled / KHÔNG còn label "Sắp ra mắt".

### AC-2 — Profile page hiển thị 5 read-only field

**Given** user ở `/profile`
**When** page render
**Then** thấy: email (readonly text), displayName (editable inline), role badge, lastLoginAt (relative time), createdAt (relative time)
**And** nếu có `avatarUrl` thì hiển thị `<img>`; nếu null fallback initials gradient.

### AC-3 — Edit displayName happy path

**Given** user ở `/profile` Profile section
**When** user click "Sửa" → đổi displayName → click "Lưu"
**Then** API `PATCH /api/v1/me {displayName}` trả 200
**And** sonner toast "Đã cập nhật hồ sơ"
**And** UserMenu trigger refresh tên mới (header avatar aria-label updated)
**And** F5 page → tên mới persist.

### AC-4 — Change password happy path

**Given** user ở `/profile` Security section
**When** điền 3 field (oldPassword, newPassword, confirmNewPassword) match + submit
**Then** API `POST /api/v1/me/password` trả 204
**And** toast "Đã đổi mật khẩu — các phiên khác đã đăng xuất"
**And** form reset 3 field rỗng
**And** session hiện tại vẫn login (không bị kick).

### AC-5 — Wrong old password 401 inline

**Given** user điền `oldPassword` sai
**When** submit
**Then** server 401 `INVALID_CREDENTIALS`
**And** form hiển thị inline error "Mật khẩu cũ không đúng" gần field oldPassword
**And** form không bị reset (giữ newPassword user vừa gõ).

### AC-6 — New password validation

**Given** user gõ `newPassword` < 8 chars hoặc 2 field không match
**When** submit
**Then** FE form hiển thị inline error trước khi POST (Zod client-side hoặc react-hook-form rule)
**And** không request lên server.

### AC-7 — Avatar upload + replace

**Given** user ở `/profile` Avatar section
**When** click upload button → chọn file `.jpg/.png/.webp` ≤ 2 MB → confirm
**Then** API `POST /api/v1/me/avatar` trả 200 với absolute Cloudinary URL
**And** ảnh hiện preview trong section + UserMenu trigger header
**And** toast "Đã cập nhật ảnh đại diện"
**And** upload lần 2 → URL mới ghi đè, cũ không cleanup (accepted drift).

### AC-8 — Avatar upload validation

**Given** user upload file > 2 MB
**When** request đến server
**Then** server 413 `FILE_TOO_LARGE` (UI toast "File quá lớn (≤ 2 MB)")
**Or** mime ≠ image → 415 `UNSUPPORTED_MEDIA_TYPE` (UI toast "File phải là ảnh PNG/JPG/WebP").

### AC-8a — Avatar delete (amend 2026-05-16)

**Given** user có avatar đã upload (`users.avatar_url` not null)
**When** user mở AvatarUploadDialog → click "Xóa ảnh" → API `DELETE /api/v1/me/avatar`
**Then** server gọi Cloudinary `destroy(publicId)` best-effort (log warn nếu fail nhưng vẫn proceed)
**And** `users.avatar_url` set về `null`
**And** response 204 No Content
**And** UI fallback về initials gradient + toast "Đã xóa ảnh đại diện"
**And** subsequent DELETE khi đã null vẫn trả 204 (idempotent).

### AC-9 — Session purge — other devices kicked

**Given** user đã login trên 2 browser (sid_A + sid_B), đang sit sid_A
**When** đổi password thành công trên sid_A
**Then** sid_B Redis key bị delete (`purgeSessionsForUserExcept(redis, userId, sid_A)`)
**And** sid_B trên request kế tiếp → 401 UNAUTHENTICATED (auto redirect login).

### AC-10 — IDOR resistance

**Given** user A login (sid_A)
**When** user A gửi `PATCH /api/v1/users/<userB-id>` (admin endpoint US-007)
**Then** vẫn 403 (US-007 admin gate). KHÔNG cần thay đổi US-007 — `/me` endpoint mới không nhận `:id` URL param.

## Risks & open items

- **Stale UserMenu avatar after mutation** — mitigate qua `qc.invalidateQueries(authKeys.me)` trong mỗi mutation hook. Test cover.
- **Race condition session purge** — user đổi password rồi click logout cùng thời điểm → có thể double-destroy current sid. Acceptable (logout idempotent).
- **Cloudinary cost** — pilot < 100 user × 100KB avatar ≈ 10 MB; free tier 25 GB → 0.04% utilization. OK.
- **No password strength meter** — chỉ enforce min 8 chars v1. Add zxcvbn nếu user yêu cầu sau.
