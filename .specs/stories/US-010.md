## User perspective

> As an **authenticated user (any role)**, I want **to fill in extended profile fields (phone, department, location, short bio)** on my `/profile` page, so that **`PersonalInfoCard` của tôi không hiển thị giá trị fake hardcoded ("0901 234 567", "Product · BA Team") nữa và teammate xem hồ sơ thấy đúng thông tin của tôi**.

## Why this story now?

ProfilePage v4.1 (CR-006 `31f8d3c`) shipped `PersonalInfoCard` với 4 InfoRow hardcoded fake strings (Họ tên / Điện thoại / Phòng ban / Địa chỉ). User feedback "kiểm tra lại toàn bộ giao diện xem những chỗ nào đang mock dữ liệu" → đây là surface hard-mock visible nhất trên Profile. Effort thấp (4 nullable cột + extend PATCH /me), value cao (xóa fake data khỏi UI).

## Scope in

- Schema: 4 nullable text columns trên `users` (`phone`, `department`, `location`, `bio`) qua migration `0010_users_profile_enrichment.sql`.
- API:
  - `GET /api/v1/me` response shape extend với 4 field (nullable strings).
  - `PATCH /api/v1/me` body extend: accept optional `phone`, `department`, `location`, `bio`. Explicit `null` clears; missing key leaves untouched.
- Shared: `AuthUser` schema + `updateProfileSchema` extend với 4 optional field + phone regex + length limits.
- FE:
  - `PersonalInfoCard` đọc `data.user.phone / department / location` thay hardcoded. Field rỗng → render dash + "Chưa cập nhật".
  - `EditProfileDialog` form extend 4 input (Phone / Department / Location / Bio textarea), validate client-side.
  - `Bio` field hiển thị dưới h1 Hồ sơ trong profile card (optional v1 — chỉ show nếu có).

## Acceptance criteria

### AC-1 — GET /me returns 4 new fields

**Given** user đã login với row có `phone='0901 234 567', department='BA Team', location='HCM', bio='hi'`
**When** GET /me
**Then** response `data.user` chứa cả 4 field với giá trị đó
**And** với user row có cả 4 = NULL → response chứa cả 4 field giá trị `null`.

### AC-2 — PATCH /me updates 4 fields

**Given** user authenticated
**When** PATCH /me body `{ phone: "0912345678", department: "Dev", location: "HN", bio: "Hello" }`
**Then** response 200 với `data.user` reflect 4 field mới
**And** F5 page → 4 field persist trong DB.

### AC-3 — PATCH /me partial update preserves other fields

**Given** user có `phone='A', department='B', location='C', bio='D'`
**When** PATCH /me body chỉ `{ phone: "X" }`
**Then** `phone` = "X"; `department/location/bio` vẫn `B/C/D` (không bị clear).

### AC-4 — PATCH /me with explicit null clears

**Given** user có `phone='A'`
**When** PATCH /me body `{ phone: null }`
**Then** DB row `phone IS NULL`.

### AC-5 — PATCH /me rejects invalid phone format

**Given** user authenticated
**When** PATCH /me body `{ phone: "abc!" }` (chứa chữ + ký tự đặc biệt ngoài allowlist)
**Then** server 400 `VALIDATION_ERROR` với `details` chỉ ra field `phone`
**And** DB row không thay đổi.

### AC-6 — PATCH /me rejects over-length bio

**Given** user authenticated
**When** PATCH /me body `{ bio: "x".repeat(501) }`
**Then** server 400 `VALIDATION_ERROR` field `bio`.

### AC-7 — PersonalInfoCard renders real fields

**Given** user ở `/profile` với row có `phone='0901', department='Dev', location='HCM'`
**When** page render
**Then** `PersonalInfoCard` row "Điện thoại" hiển thị `0901` (thay vì `0901 234 567` hardcoded cũ)
**And** row "Phòng ban" hiển thị `Dev` (thay vì `Product · BA Team` hardcoded cũ)
**And** row "Địa chỉ" hiển thị `HCM` (thay vì `TP. Hồ Chí Minh` hardcoded cũ).

### AC-8 — PersonalInfoCard shows "Chưa cập nhật" placeholder

**Given** user có row với 4 field = NULL
**When** page render
**Then** 3 row (Điện thoại / Phòng ban / Địa chỉ) hiển thị dash + muted text "Chưa cập nhật"
**And** row "Họ tên" vẫn render `displayName` (luôn có giá trị, không null-able).

### AC-9 — EditProfileDialog extends form

**Given** user click "Cập nhật hồ sơ" → dialog mở
**When** điền 4 input + click "Lưu"
**Then** PATCH /me sent với 4 field
**And** toast "Đã cập nhật hồ sơ"
**And** dialog close
**And** `PersonalInfoCard` refresh ngay (qua invalidate `authKeys.me`).

## UX notes

## Risks & open items

- **Phone format strictness** — `/^[0-9 +\-()]{1,30}$/` không validate semantic (length + country code). User compromise: accept loose v1, tighten nếu cần.
- **Bio sanitization** — render plain text trên FE, không markdown; XSS-safe nhờ React default escaping.
- **Department free-text drift** — different users sẽ gõ "BA", "BA Team", "Business Analysis". Acceptable v1; future v2 có thể normalize hoặc thêm dictionary.
