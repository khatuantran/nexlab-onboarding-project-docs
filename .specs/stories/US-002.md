## User perspective

> As a **BA/PM (Lan)**, I want to **create a new project and add a feature với business/user-flow/business-rules filled**, so that **devs có authoritative source of truth trước khi kickoff sprint**.

## Why after US-001?

US-001 đã có read path + auth + DB schema + layout. US-002 chỉ thêm write flow (forms + PUT/POST endpoints) lên foundation có sẵn. Tránh YAGNI authoring UI trước khi reader render được.

---

## Gate 0 decisions (approved 2026-04-23)

Pre-UX-spec decisions driving scope:

1. **Modal lib**: Radix Dialog + shadcn wrapper (`@radix-ui/react-dialog` + `components/ui/dialog.tsx`). Zero Vite-quirk, a11y built-in.
2. **Section editor layout**: 2-col split (textarea | preview) desktop, stack dọc mobile. Realtime preview debounced 200ms.
3. **Edit route**: same `/projects/:slug/features/:featureSlug` (US-001), không tách `/edit`. Toggle view ↔ edit per-section.
4. **Toast**: `sonner` — success 2s auto-dismiss, destructive 5s + retry action.

---

## Scope in

- Admin UI: nút "Tạo project" trong header (chỉ hiển thị với role `admin`).
- Form tạo project: input `slug` (auto-kebab từ name), `name`, `description` (textarea, optional).
- Form tạo feature trong project: input `slug`, `title`. Submit → auto-init 5 section empty → redirect về feature detail page (same route, edit-enabled).
- Feature detail page `/projects/:slug/features/:featureSlug` (từ US-001) thêm **edit-in-place**: mỗi section card có nút "Sửa" (admin/author only) toggle view ↔ edit mode. Không route riêng.
- Section editor cho 3 business sections (`business`, `user-flow`, `business-rules`): markdown textarea bên trái + live preview bên phải (2-col split), nút "Lưu" / "Hủy" explicit. Mỗi section toggle edit độc lập.
- Hiển thị metadata "Updated by <name>, <relative time>" cho từng section đã có update.

## Scope out

- Edit `tech-notes` và `screenshots` → US-003.
- Upload file → US-003.
- Delete project / feature / section → v2.
- Autosave → explicit save only v1.
- Optimistic concurrency (last-write-wins).
- Rich text WYSIWYG (markdown thuần).
- Embed preview rendering (sẽ xuất hiện ở US-003 và feature detail nếu BA paste link).

## Acceptance criteria (Given/When/Then)

### AC-1 — Admin gate cho tạo project

**Given** Lan login với role `admin`
**When** Lan load trang bất kỳ
**Then** header hiển thị button "Tạo project"
**And** khi user role `author` (không admin) login, button không xuất hiện.
**And** truy cập trực tiếp `POST /api/v1/projects` với user không phải admin → 403 `FORBIDDEN`.

### AC-2 — Tạo project happy path

**Given** Lan là admin, ở form "Tạo project"
**When** Lan nhập `name="Pilot Project"`, slug auto-fill `pilot-project`, description optional, submit
**Then** API trả 201 với `{ data: { id, slug, name, createdAt } }`
**And** browser redirect về `/projects/pilot-project` (landing rỗng)
**And** trang landing hiển thị "Chưa có feature nào" (reuse empty state từ US-001 AC-4).

### AC-3 — Slug duplicate

**Given** project `pilot-project` đã tồn tại
**When** Lan submit form với slug `pilot-project`
**Then** API trả 409 `PROJECT_SLUG_TAKEN`
**And** UI hiển thị inline error cạnh input slug: "Slug đã được dùng, chọn slug khác".
**And** form không reset các input khác.

### AC-4 — Tạo feature trong project

**Given** Lan ở `/projects/pilot-project` và click "Thêm feature"
**When** Lan nhập `title="Đăng nhập bằng email"` + slug auto `dang-nhap-bang-email` và submit
**Then** API `POST /api/v1/projects/pilot-project/features` trả 201
**And** 5 section rows tạo đồng thời trong DB (transaction atomic)
**And** browser redirect về `/projects/pilot-project/features/dang-nhap-bang-email` (same route như view)
**And** 5 section card hiển thị empty với button "Sửa" + placeholder "Chưa có nội dung" (nếu role = admin/author).

### AC-5 — Edit business section

**Given** Lan ở feature detail page với section `business` empty, role = admin
**When** Lan click "Sửa" ở section `business`, gõ markdown, click "Lưu"
**Then** API `PUT /api/v1/features/:id/sections/business` trả 200 với body đã lưu
**And** section card switch từ editor về read-mode ngay (2-col collapse → markdown render)
**And** sonner toast "Đã lưu" hiển thị 2s
**And** metadata "Updated by Lan, vài giây trước" hiển thị dưới heading section.
**And** refresh trang → content persist, metadata vẫn đúng.

### AC-6 — Save 2 section độc lập

**Given** Lan đã edit section `business`
**When** Lan edit section `user-flow` và save (mà không động section `business`)
**Then** `business.updated_at` không thay đổi
**And** `user-flow.updated_at` là thời gian save mới
**And** feature.updated_at refresh (do có section con đổi) → project landing hiển thị feature này ở top (AC-3 US-001).

### AC-7 — Validation markdown quá lớn

**Given** Lan paste 100 KB markdown vào section editor
**When** Lan click Save
**Then** API trả 413 `SECTION_TOO_LARGE` (limit 64 KiB)
**And** sonner toast destructive "Nội dung section quá lớn (>64 KiB)" + giữ nguyên draft trong editor để Lan tỉa.

### AC-8 — Unauthenticated / wrong user

**Given** không có session hoặc session expired
**When** Lan PUT section
**Then** API trả 401
**And** FE redirect về `/login?next=<current>`.

## UX notes (wire-level)

- **Create project**: Radix Dialog (shadcn wrapper) overlay từ AppHeader "Tạo project" button (admin only).
- **Create feature**: Radix Dialog từ project landing "Thêm feature" button (admin/author).
- **Feature detail page (US-001)**: thêm nút "Sửa" góc phải mỗi section card (admin/author only). Click → card expand thành 2-col (textarea trái + preview phải), footer "Lưu" / "Hủy". Click "Hủy" revert draft + collapse. Per-section state độc lập (có thể 2 section edit cùng lúc).
- **Markdown preview**: render realtime khi gõ (debounce 200ms), dùng cùng sanitize pipeline với read view (reuse `MarkdownView` từ T9).
- **Toast**: sonner — success "Đã lưu" (2s auto-dismiss), destructive cho lỗi (5s, action "Thử lại").
- **"Relative time"**: reuse `RelativeTime` từ US-001 T9 (`date-fns/formatDistanceToNow` locale `vi`).

## Risks & open items

- **Role column**: nếu T3 của US-001 đã schema users có `role`, US-002 zero migration. Nếu chưa, cần 1 migration bổ sung — flag khi task kickoff.
- **Markdown preview performance**: render realtime 64 KiB → có thể lag. Debounce 200ms nếu chậm.
- **Concurrency**: 2 user cùng edit section `business` → last-write-wins, documented trong FR-FEAT-003. Không hiển thị warning v1 (accepted trade-off).
