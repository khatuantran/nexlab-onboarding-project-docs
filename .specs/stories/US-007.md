## User perspective

### As an Admin

> As an **Admin**, I want **trang `/admin/users` để xem toàn bộ tài khoản, mời người mới, đổi role, disable hoặc reset mật khẩu**, so that **tôi không phải SSH vào DB chạy SQL khi onboard dev mới hoặc thu hồi access**.

## Why this story now?

- Pilot đang chạy với seed admin (`admin@local`) + manual SQL invite (xem [02-requirements §Open issues #4](../02-requirements.md)). Không scale khi M3 mời ≥ 3 dev mới onboard.
- Khi nhân sự nghỉ, cần disable account ngay; không có UI → SSH + DB UPDATE, blast radius cao + không audit.
- M4 roadmap đã list "Admin UI quản lý user" làm candidate; story này promote candidate → committed slice (full: list + invite + edit + reset).

## Scope in

- Trang `/admin/users` (admin-only) hiển thị table users (avatar, displayName, email, role, lastLogin, status).
- Filter: free-text query (display_name / email), role, status (active / archived).
- Invite dialog: nhập email + displayName + role → BE tạo user với password tạm random, trả lại 1 lần cho admin copy.
- Edit dialog: đổi displayName, role.
- Action menu per row: Edit / Disable (archive) / Re-enable (unarchive) / Reset password.
- Disable user: login `POST /auth/login` reject với 403 `USER_DISABLED`.
- Reset password: admin trigger → BE sinh password tạm mới, trả lại 1 lần cho admin copy.
- Self-service password change: deferred (xem Scope out) — pilot dùng admin reset.

## Scope out

- Email/SMTP integration → reset / invite gửi email tự động. V1 admin copy-paste link / password thủ công.
- Self-service password change endpoint (`POST /auth/me/password`). Đẩy backlog (BL-NNN sau pilot retro).
- Role mới (viewer / read-only). V1 vẫn 2 role admin/author.
- SSO / OAuth.
- Pagination > 50 user. V1 limit 100 hard, sort displayName asc.
- Audit log per action (chỉ log `pino info` không lưu DB).

## Acceptance criteria

### AC-1 — Non-admin không xem được trang

**Given** tôi đăng nhập với role `author`
**When** tôi navigate tới `/admin/users`
**Then** UI redirect về `/` hoặc render "Bạn không có quyền truy cập" (admin gate giống CreateProjectDialog).
**And** `GET /api/v1/users?detail=admin` trả `403 FORBIDDEN`.

### AC-2 — Admin list + filter

**Given** tôi là admin
**When** tôi mở `/admin/users` và gõ `q=lan` + chọn role=author
**Then** table chỉ hiển thị user có displayName / email chứa "lan" và role=author.
**And** mỗi row hiển thị avatar, displayName, email, role, lastLogin (relative time hoặc "—"), status badge (Active / Disabled).

### AC-3 — Admin invite user mới

**Given** tôi là admin, click "Mời user mới"
**When** tôi nhập email `dev@nexlab.com`, displayName `Tâm`, role `author` và submit
**Then** BE tạo user với password tạm 12 ký tự random.
**And** response trả `{ data: { user: {...}, tempPassword: "..." } }` đúng 1 lần.
**And** modal hiển thị tempPassword với nút Copy + cảnh báo "Không lưu lại sau khi đóng".
**And** user mới xuất hiện trong table sau refresh / mutation invalidate.

### AC-4 — Email trùng → 409

**Given** đã có user `dev@nexlab.com`
**When** admin invite user khác cùng email
**Then** BE trả `409 USER_EMAIL_EXISTS`.
**And** dialog hiển thị error message tiếng Việt.

### AC-5 — Edit displayName + role

**Given** admin click Edit trên row user X (role=author)
**When** đổi role → admin và submit
**Then** `PATCH /users/:id` trả `200 { data: User }` với role mới.
**And** table refresh, badge role cập nhật.
**And** không cho phép admin tự đổi role chính mình (BE 403 `CANNOT_MODIFY_SELF` để tránh khoá account cuối cùng — xem Risks).

### AC-6 — Disable user → login reject

**Given** admin click Disable trên row user X
**When** confirm dialog
**Then** `POST /users/:id/archive` set `archived_at = NOW()`.
**And** sau đó user X thử login → `403 USER_DISABLED` (FE login page hiện copy "Tài khoản đã bị vô hiệu hoá, liên hệ admin").
**And** Re-enable trả về `archived_at = NULL` qua `POST /users/:id/unarchive`.

### AC-7 — Reset password

**Given** admin click "Reset mật khẩu" trên row user X
**When** confirm dialog
**Then** `POST /users/:id/reset-password` sinh tempPassword mới 12 ký tự, hash + store, trả về plaintext 1 lần.
**And** UI hiển thị modal copy tempPassword (giống AC-3).
**And** session hiện có của user X bị invalidate (Redis flush by userId) — user X login lại bằng tempPassword.

### AC-8 — Self-protect: admin cuối cùng không bị disable

**Given** chỉ còn 1 admin trong hệ thống (chính tôi)
**When** tôi cố disable account của mình HOẶC đổi role mình về author
**Then** BE trả `409 LAST_ADMIN_PROTECTED`.
**And** UI hiển thị error message rõ.

## Risks & open items

- **Last-admin lockout**: nếu chỉ còn 1 admin và self-disable, không ai phục hồi được. → AC-8 `LAST_ADMIN_PROTECTED` mitigate.
- **Temp password phơi qua admin**: admin thấy plaintext 1 lần. Acceptable vì admin = trusted role; alternative (email link) defer scope out.
- **Session invalidation race**: nếu user X đang có session khi bị reset/disable, request giữa lúc archived_at set có thể vẫn pass. → middleware check `archived_at` mỗi request (xem T4).
- **Bcrypt cost cho temp password**: dùng cùng cost factor như register flow (12). Reset endpoint tạo password mới phải hash sync; pilot scale OK.

## Test plan

- **Unit (repo)**: `userRepo` thêm: `createUser({email, displayName, role, passwordHash})`, `updateUser(id, patch)`, `archiveUser(id)`, `unarchiveUser(id)`, `countActiveAdmins()`, `getUserById(id)`. Test happy + duplicate + last-admin guard.
- **Unit (helper)**: `generateTempPassword()` returns 12-char string với charset alphanumeric (no ambiguous I/l/0/O), high entropy.
- **Integration (route)**: 5 endpoint cover happy + 401 + 403 (non-admin) + 409 duplicate / last-admin + 404 not found.
- **Integration (login)**: archived user login → 403 `USER_DISABLED`.
- **E2E (Playwright `e2e/us-007.spec.ts`)**: admin login → /admin/users → invite (copy temp pass) → edit role → disable → re-enable → reset password → logout → login as new user with temp pass.
