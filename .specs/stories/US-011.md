## User perspective

> As a **user browsing projects + features**, I want **the contributor avatars trên ProjectCard / ProjectHero / FeatureCard / ActivityRail show real editors** (derived from who actually edited sections), so that **tôi không bị nhìn nhầm "TM/NL" hardcoded khi project thực có author khác hoặc rỗng**.

## Why this story now?

CR-006 v4 UI redesign shipped 4 surfaces với hardcoded contributor data:

- `ProjectCard.tsx` `CONTRIBUTOR_POOL = ["EK","PT","KT","NL","TR","TM"]` + `pickContributors(slug, 3)` hash-derived.
- `ProjectHero.tsx` `AvatarStack names={["TM","NL","PT","HD"]}` 4 fixed initials.
- `FeatureCard.tsx` `const contributors = ["TM","NL"]` cố định.
- `ActivityRail.tsx` `STATIC_PADDING` 2 dummy activity rows.

User feedback "kiểm tra lại toàn bộ giao diện xem những chỗ nào đang mock dữ liệu" — đây là surface mock visible nhất khắp app (mọi project card đều fake contributors).

`sections.updated_by` đã FK đến `users.id` (US-002 schema). Đủ derive top contributors per scope mà không cần membership table.

---

## Scope in

- DB: zero schema change. Reuse `sections.updated_by` + `sections.updated_at`.
- API: extend response shape (additive, backward-compatible):
  - `GET /api/v1/projects` → mỗi item bổ sung `contributors: ContributorSummary[]` (max 5).
  - `GET /api/v1/projects/:slug` → `data.project.contributors` (max 5).
  - `GET /api/v1/projects/:slug/features/:featureSlug` → `data.feature.contributors` (max 5).
- Repo:
  - `projectRepo.getContributorsForProject(projectId, limit=5)` — distinct `sections.updated_by` joined `users`, filter `features.archived_at IS NULL`, sort by latest `sections.updated_at` desc.
  - `projectRepo.getContributorsForFeature(featureId, limit=5)` — same pattern, no archive filter (feature scope already-known active).
  - Batched variant for list endpoint to avoid N+1.
- Shared: new `ContributorSummary` Zod schema (`{ userId, displayName, avatarUrl, lastUpdatedAt }`); extend `ProjectSummary` / `ProjectResponse` / `FeatureResponse`.
- FE — drop all 4 hardcoded sites:
  - `ProjectCard` reads `project.contributors`; remove `CONTRIBUTOR_POOL` + `pickContributors`.
  - `ProjectHero` accepts `contributors` prop; ProjectLandingPage passes through.
  - `FeatureCard` reads `feature.contributors`.
  - `ActivityRail` drops `STATIC_PADDING`; renders only real `sections.updated_*` projection.
- `AvatarStack`: extend to accept `contributors: ContributorSummary[]` variant (rendering avatarUrl when present, fallback initials from displayName).

## Scope out

- **Membership / explicit contributor assignment** — derived only v1.
- **Contributor pagination / "see all"** — top 5 cap only.
- **Real-time updates** — fetched on read; no WebSocket.
- **Contributor weight by edit volume** — sorted by recency only.
- **Audit log persistence** — already deferred (FR-DATA-001 only pino).
- **Avatar image rendering in AvatarStack** beyond what already exists — if AvatarStack only supports initials today, that's fine (initials from real displayName already better than fake "TM/NL"); image render is nice-to-have.
- **Feature edit endpoint exposing contributors** — only read endpoints augment.

## Acceptance criteria

### AC-1 — GET /projects returns contributors per project

**Given** 3 projects (`a`, `b`, `c`), trong đó project `a` có 3 user (`u1`, `u2`, `u3`) edit sections gần đây (u3 most recent, u1 oldest)
**When** GET /projects
**Then** project `a` row có `contributors` array length 3, sorted [u3, u2, u1] by `lastUpdatedAt` desc
**And** mỗi entry có `{userId, displayName, avatarUrl, lastUpdatedAt}`.

### AC-2 — Contributors capped at 5

**Given** project có 7 distinct editors
**When** GET /projects/:slug
**Then** `data.project.contributors.length === 5` (top 5 by recency).

### AC-3 — Brand-new project → empty contributors

**Given** project mới tạo, chưa có feature, sections rỗng (no `updated_by` set)
**When** GET /projects/:slug
**Then** `contributors: []`.

### AC-4 — Archived features excluded from project contributors

**Given** project có 1 feature archived (edited by `u1` only) + 1 feature active (edited by `u2`)
**When** GET /projects/:slug
**Then** `contributors` chỉ chứa `u2` (không có `u1`).

### AC-5 — Feature scope contributors

**Given** feature có 2 user edit sections (`u1`, `u2`); cùng project có feature khác edited by `u3`
**When** GET /projects/:slug/features/:featureSlug
**Then** `data.feature.contributors` chứa chỉ `u1` + `u2` (không có `u3`).

### AC-6 — ProjectCard renders real avatars

**Given** HomePage load, project có contributors `[Tester, Lan, Minh]`
**When** card render
**Then** `AvatarStack` render initials `T / L / M` (derived from real displayName), không phải `EK/PT/KT` hash-derived.
**And** project rỗng contributors → AvatarStack render nothing (hoặc empty placeholder).

### AC-7 — ProjectHero + FeatureCard wire real data

**Given** ProjectLandingPage + FeatureCard render
**When** project có contributors thực
**Then** không còn `["TM","NL","PT","HD"]` literal hardcoded
**And** stack render initials từ `project.contributors[i].displayName`.

### AC-8 — ActivityRail drops STATIC_PADDING

**Given** FeatureDetailPage render
**When** feature có 0-1 sections edited
**Then** ActivityRail chỉ render real `sections.updated_*` rows (không pad 2 fake row "Ngọc Linh / Phương Trâm").

### AC-9 — Performance: list endpoint no N+1

**Given** GET /projects với N=20 projects
**When** request handled
**Then** SQL log show 1 query for projects + 1 batched query for contributors (≤ 2 total), not N+1.

## Risks & open items

- **N+1 perf**: must batch the contributors lookup in `/projects` list. Use single SQL with `DISTINCT ON` + `ROW_NUMBER()` window OR Drizzle's `with` clause. Profile estimate < 30 ms for N=20.
- **Avatar rendering**: AvatarStack today might only accept `names: string[]`. Need to extend with `contributors: ContributorSummary[]` variant OR continue passing displayName array. Decision: pass `string[]` initials derived client-side (simpler), defer avatarUrl rendering to BL-NNN.
- **Soft-deleted users**: if a user was archived (US-007), `users.archived_at != null` but their past `updated_by` rows still reference them. Decision: include them (display "Archived User" or keep displayName) — `users` join via FK `set null` means archived user data lost on hard delete only; archived user row still exists.
- **No `updated_by` for content seed** — initial seed sections may have `updated_by = NULL` → exclude from join (LEFT JOIN với `WHERE u.id IS NOT NULL`).
