## User perspective

### As a Reader (Dev mới onboard)

> As a **Reader**, I want **search trả về kết quả nhóm theo project / feature / section / author / file**, so that **tôi nhận ra ngay loại match và đi đúng nơi cần đến mà không quét hết list flat**.

### As a BA / PM

> As a **BA**, I want **filter section-type "user-flow" để tìm business flow cụ thể giữa các feature**, so that **tôi đối chiếu nghiệp vụ giữa các feature liên quan mà không cần mở từng feature**.

### As a Senior Dev

> As a **Tech Lead**, I want **filter theo author + thời gian cập nhật**, so that **tôi review nhanh ai vừa edit cái gì gần đây mà không vào git history**.

US-001 ship search v1 (FR-SEARCH-001) chỉ search feature title + aggregate section bodies, kết quả flat. Sau khi M2 + CR-002 ship (5 screens UI uplift), corpus pilot bắt đầu tăng → user feedback 2026-04-25: muốn search rộng hơn (project name, ai đang viết, screenshot), kèm filter "business flow" (section-type `user-flow`). Đây là capability mở rộng (không phải fix US-001) → US-005 mới.

## Scope in

- **Backend**:
  - DB migration thêm `tsvector` cho `projects` + `uploads` + cột `uploads.caption text` optional. GIN indexes.
  - Repo method `searchAll(q, opts)` chạy 5 query song song, mỗi entity trả top-5 ranked.
  - Route `GET /api/v1/search` extend query params (`sectionTypes`, `authorId`, `updatedSince`, `status`); response shape thay từ flat `data: SearchHit[]` → grouped `data: { projects, features, sections, authors, uploads }` (breaking).
  - Endpoint mới `GET /api/v1/users?q=&role=` cho author filter dropdown.
  - Archived projects loại khỏi mọi group.
- **Frontend**:
  - SearchPage rewrite layout grouped (5 group sections, conditional render khi có hit).
  - FilterBar component composite: SectionTypeChips (multi) + AuthorPicker (combobox + `useUsers`) + TimeRangeDropdown (preset) + StatusChips.
  - URL state: filters serialize vào query string, only non-default values.
  - Per-entity result cards: ProjectResultCard, FeatureResultCard (rename + refactor existing SearchResultRow), SectionResultCard, AuthorResultCard, UploadResultCard.
  - Section deep-link click → `/projects/:projectSlug/features/:featureSlug#section-{type}`. Verify FeatureDetailPage anchor id `section-{type}` (fix nếu chưa chuẩn hoá).
  - Idle state + zero-result state copy update theo filters active.
- **Shared**: `packages/shared/src/search.ts` — `SearchResultsV2` Zod schema + per-hit shapes + `SearchOpts` filter types.
- **Tests**: unit BE per-entity + filter combos, integration route + filters, unit FE filter chip toggle + URL roundtrip + grouped render gating, E2E happy-path full search → filter → click section deep-link.

## Acceptance criteria

### AC-1 — Grouped response shape

**Given** authenticated user submits `GET /api/v1/search?q=login`
**When** server processes request
**Then** response body = `{ data: { projects, features, sections, authors, uploads } }`
**And** mọi key xuất hiện kể cả khi array rỗng
**And** mỗi group ≤ 5 hits, sorted by `rank` desc.

### AC-2 — Project hit từ name match

**Given** project có `name = "Pilot Đăng nhập"`, search query `q=đăng nhập`
**When** user submit
**Then** `data.projects[0].slug` match project đó
**And** `snippet` chứa `<mark>` quanh "Đăng nhập".

### AC-3 — Section hit deep-link

**Given** feature `pilot/login` có section `user-flow` body chứa "OTP gửi qua SMS"
**When** user search `q=OTP`
**Then** `data.sections[0]` chứa `{ projectSlug: "pilot", featureSlug: "login", sectionType: "user-flow", snippet }`
**And** FE click section card → navigate `/projects/pilot/features/login#section-user-flow`
**And** browser scroll vào section anchor.

### AC-4 — Author hit count

**Given** user "Lan" đã edit 3 distinct features
**When** search `q=lan`
**Then** `data.authors[0]` chứa `{ displayName: "Lan", touchedFeatureCount: 3 }`.

### AC-5 — Upload hit qua filename

**Given** upload `filename="login-flow.png"` thuộc feature `pilot/login`
**When** search `q=login-flow`
**Then** `data.uploads[0]` chứa `{ filename: "login-flow.png", projectSlug: "pilot", featureSlug: "login" }`.

### AC-6 — Filter section-type narrows

**Given** corpus có 10 section match query, 4 thuộc type `user-flow`
**When** user search với `sectionTypes=user-flow`
**Then** `data.sections.length ≤ 5` và mọi hit có `sectionType=user-flow`
**And** `data.features` chỉ chứa feature có ít nhất 1 section type khớp.

### AC-7 — Filter author narrows

**Given** user `authorId=<UUID Lan>` đã edit 2 sections match query
**When** user search với `authorId=<UUID>`
**Then** `data.sections` chỉ chứa hits do Lan touch
**And** `data.uploads` chỉ chứa uploads do Lan upload.

### AC-8 — Filter time-window narrows

**Given** 5 features match query, 2 trong số đó `updated_at` < 7 ngày
**When** user search với `updatedSince=<ISO 7d ago>`
**Then** `data.features` chỉ chứa 2 feature đó.

### AC-9 — Filter status narrows

**Given** 3 features match query, 1 có 5/5 sections filled
**When** user search với `status=filled`
**Then** `data.features` chỉ chứa feature đó.

### AC-10 — Archived project loại khỏi mọi group

**Given** project `old-pilot` archived, có feature match query
**When** user search query bất kỳ
**Then** mọi entity group **không** chứa hit thuộc `old-pilot`.

### AC-11 — User list endpoint

**Given** authenticated user submit `GET /api/v1/users?q=lan&role=author`
**When** server processes
**Then** response `{ data: User[] }` với `User = { id, displayName, role }`
**And** không có field `email` / `passwordHash`
**And** sorted `displayName` asc, limit 50.

### AC-12 — FE grouped render gating

**Given** search result có 0 hits trong `uploads` group
**When** SearchPage render
**Then** group "Uploads" header không xuất hiện
**And** chỉ render 4 group còn lại (theo thứ tự: Projects → Features → Sections → Authors → Uploads, skip empty).

### AC-13 — FE filter URL state roundtrip

**Given** user toggle SectionType `user-flow` + chọn author Lan
**When** browser address bar update
**Then** URL chứa `?q=...&sectionTypes=user-flow&authorId=<UUID>`
**And** copy URL, paste tab mới → SearchPage re-hydrate filter state đúng + refetch.

### AC-14 — FE zero-result với filter active

**Given** query có hit khi không filter, nhưng filter `status=filled` ra 0
**When** SearchPage render
**Then** zero-result panel hiện copy "Không có kết quả với filter hiện tại"
**And** liệt kê filter active dạng chip removable
**And** click chip × → remove filter → refetch.

### AC-15 — XSS sanitize snippet (regression FR-SEARCH-001)

**Given** section body chứa `<script>alert(1)</script>` + match keyword
**When** user search → SectionResultCard render snippet
**Then** snippet không thực thi script
**And** chỉ `<mark>` được giữ; mọi tag khác strip.

## Risks & open items

- **Breaking response shape**: `SearchHit[]` → `SearchResultsV2`. Mọi caller (`useSearch` hook, AppHeader behavior) phải migrate cùng PR atomic. Mitigation: T6 + T9 cùng commit; tests bao trùm round-trip.
- **Per-entity query × 5 latency**: target ≤ 500ms p95 (NFR-PERF-001). Mitigation: `Promise.all`, GIN indexes per entity, benchmark seeded 100 features < 200ms p95 mục tiêu.
- **Author endpoint exposure**: trả `displayName + role` cho mọi authenticated user — chấp nhận v1 (FR-PROJ-001 access model). Document risk; nếu pilot mở external → revisit.
- **URL length explosion**: 4 filter dimensions có thể >2k chars. Mitigation: only serialize non-default values; status enum 1-char compress nếu cần.
- **Section anchor id**: FeatureDetailPage hiện tại có thể chưa render `id="section-<type>"` trên section heading. T9 verify + fix nếu cần (in-scope, không spawn task riêng).
- **Snippet duplicate UX**: cùng query có thể match cả Feature group (snippet aggregate) lẫn Section group (snippet riêng). User nhìn duplicate. Mitigation: feature snippet aggregate giữ behavior cũ; UX charter §3 group header đủ phân biệt; document trade-off.
