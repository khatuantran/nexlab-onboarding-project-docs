## Summary

As an **authenticated user**, I want **to "Theo dõi" (watch) projects and features** so that **the buttons that currently fire `placeholderToast` actually persist a watch relationship + (eventually) feed BL-008 notifications**.

## Motivation

CR-006 v4 ships "Theo dõi" buttons on both ProjectLandingPage (`placeholderToast("Theo dõi project")`) and FeatureDetailPage (`placeholderToast("Theo dõi feature")`). Dead buttons in v4 chrome. Foundational for notifications (BL-008) — watch sets the "who cares" set.

## Proposed scope

**In scope**:

- Schema: new table `watchers (user_id UUID FK, scope_type TEXT IN ('project','feature'), scope_id UUID, created_at TIMESTAMPTZ, PRIMARY KEY(user_id, scope_type, scope_id))`.
- API: `POST /api/v1/projects/:slug/watch` + `DELETE`; `POST /api/v1/projects/:slug/features/:fSlug/watch` + `DELETE`. Both idempotent.
- API: `GET /api/v1/projects/:slug` + `GET /api/v1/projects/:slug/features/:fSlug` augment response with `watchedByMe: boolean`.
- FE: button toggles between "Theo dõi" + "Đang theo dõi" (filled icon variant), optimistic update, toast on success.

**Out of scope**:

- Notification emission (BL-008 handles).
- Watch list aggregation page (separate BL if requested).
- Auto-watch on edit (defer — user explicit watch only).
- Watch counts visible to others.

## Acceptance hint

- **Given** user clicks "Theo dõi" on project, **When** POST succeeds, **Then** button label flips to "Đang theo dõi" + icon filled + idempotent on second click.
- **Given** user clicks again, **When** DELETE succeeds, **Then** button reverts.
- **Given** user reloads page, **When** GET response includes `watchedByMe: true`, **Then** button persists in "Đang theo dõi" state.

## Dependencies

- **Blocked by**: nothing.
- **Blocks**: BL-008 notification system (depends on watchers set).
