## Summary

As an **admin**, I want **the `/admin/settings` page panels (Tổng quát / Phân quyền / Thông báo / Bảo mật / Tích hợp / Giao diện) to actually persist + apply system-wide config**, so that **the entire AdminSettingsPage scaffold (currently hardcoded toggles + selects that don't save anything per `pages/AdminSettingsPage.tsx:310` comment) becomes real configuration**.

## Motivation

CR-006 v4 added AdminSettings as a placeholder admin entry point (`1a6c5d6`). Comment in source explicitly flags "hardcoded visual placeholders; toggles + selects don't persist". 6 panels worth of UI built but every interaction is no-op or toast. Largest single mock-surface in the audit. Triaged P3 — settings utility is admin-only + pilot scale doesn't need it.

## Proposed scope

**In scope** (when promoted — likely split into 4-5 stories):

- Schema: `system_settings (key TEXT PK, value JSONB, updated_at TIMESTAMPTZ, updated_by UUID FK)` — single source of truth for all settings.
- API: `GET /api/v1/admin/settings` (returns full map) + `PUT /api/v1/admin/settings` (admin-only, partial replace).
- Per-panel wiring (split):
  - **Tổng quát**: registration toggle, approval-required toggle, welcome page toggle, language, timezone.
  - **Phân quyền**: per-role permission matrix (re-grant default permissions, role rename). Heavy — could be its own story.
  - **Thông báo**: depends on BL-008 (notification system).
  - **Bảo mật**: session timeout, password policy (min length, expiry). MFA toggle (out of scope; flag only).
  - **Tích hợp**: GitHub / Jira / Slack / Google SSO connect flows. Each integration is its own backlog/spec.
  - **Giao diện**: theme defaults, logo override.
- "Cancel" / "Save" buttons actually persist via PUT.

**Out of scope** (v1 if promoted):

- Audit log per setting change (acceptable v1).
- Setting-level RBAC (admin = full; defer per-key gates).
- Real OAuth flows for integrations — likely promote to standalone stories.

## Acceptance hint
