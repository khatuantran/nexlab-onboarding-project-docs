## Summary

As a **user on HomePage**, I want **the 4 hero stat tiles + "Đủ doc" filter chip to show real numbers** instead of hardcoded `"Contributors=8"` / `"Onboard TB=2.3h"` and the placeholder `featureCount > 0` proxy for "doc-full", so that **the workspace hero gives me a truthful pulse signal**.

## Motivation

HomePage `pages/HomePage.tsx:144-151` hardcodes stat tile values. Code comment also flags the "doc" filter chip uses `result.filter((p) => p.featureCount > 0)` as a placeholder pending real `filledSectionCount`. Both surfaced in CR-006 v4.6 mock audit. P1 because hero is the first thing every user sees.

## Proposed scope

**In scope**:

- BE: `GET /api/v1/workspace/stats` returning `{ projectCount, featuresDocumented, contributorsActive, onboardingMedianHours }`.
  - `featuresDocumented` = COUNT(features) where 5/5 sections non-empty.
  - `contributorsActive` = DISTINCT `sections.updated_by` in last 30 days.
  - `onboardingMedianHours` = TBD (no signal in DB — could be hardcoded estimate or dropped).
- BE: extend `GET /api/v1/projects` `ProjectSummary` with `filledSectionCount: int` (sum of non-empty sections across features in the project).
- FE: HomePage 4 stat tiles read `useWorkspaceStats()`. Filter chip "Đủ doc" uses `project.filledSectionCount === project.featureCount * 5` instead of feature-count proxy.

**Out of scope**:

- Time-series stat (sparkline / week-over-week).
- Per-project hero stats already covered (`ProjectHero` derives).
- Real onboarding-time tracking — needs feature flag survey, defer.

## Acceptance hint

- **Given** workspace has 5 projects with 12 total features fully-documented, **When** GET /workspace/stats, **Then** `featuresDocumented === 12`.
- **Given** "Đủ doc" filter chip clicked, **When** filter applies, **Then** only projects with ALL features 5/5 remain (not just non-zero feature count).
- **Given** stats endpoint slow (>200ms), **When** HomePage mounts, **Then** stat tiles show skeleton then hydrate.

## Dependencies
