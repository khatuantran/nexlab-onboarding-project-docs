## Reproduction

1. Login vào prod FE (Netlify).
2. Mở 1 feature bất kỳ, edit section `screenshots`.
3. Click "Upload ảnh" → chọn PNG/JPG nhỏ → upload thành công, response 201 với `data.url = "/api/v1/uploads/<uuid>"`.
4. Section preview render markdown — image hiện **broken icon**, alt text fallback (e.g. `IMG_3378`).
5. DevTools Network: GET `https://nexlab-onboarding-portal.netlify.app/api/v1/uploads/<uuid>` → 200 nhưng content-type là `text/html` (SPA fallback `index.html`).

**Prerequisites**: prod deployment per CR-003 (FE và BE khác origin), user logged in.

**Frequency**: always.

## Expected vs Actual

- **Expected**: `<img>` trong markdown preview load đúng file binary đã upload, hiển thị ảnh.
- **Actual**: Browser nhận lại HTML index.html (Netlify SPA fallback) thay vì image bytes → render broken image icon.
- **Spec reference**: [FR-UPLOAD-001](../02-requirements.md#fr-upload-001--image-upload-for-screenshots) acceptance hint "Static serve qua `GET /uploads/:id`".

## Scope

- **Affected files**:
  - [apps/api/src/routes/uploads.ts](../../apps/api/src/routes/uploads.ts) — POST handler returns relative URL `/api/v1/uploads/${row.id}` (line 121); GET handler gates with `requireAuth` (line 175).
  - [apps/web/src/components/features/UploadButton.tsx](../../apps/web/src/components/features/UploadButton.tsx) — embeds `up.url` (relative) into markdown body (line 30).
  - [apps/web/src/lib/markdown.ts](../../apps/web/src/lib/markdown.ts) — sanitizer allows relative `/api/v1/uploads/...` src as-is (line 90-95); no prefix to `VITE_API_BASE_URL`.
  - [netlify.toml](../../netlify.toml) — SPA fallback `[[redirects]] /* → /index.html status=200` catches `/api/v1/*` paths and serves index.html.
- **Affected users**: every authenticated user authoring or reading any feature whose `screenshots` / `tech-notes` body has an inline upload image. All prod traffic.
- **Data corruption**: no — files persist on Fly volume + DB rows intact; only the rendering URL is wrong.
- **Rollback required**: no — pure FE rewrite + BE auth-policy change; backward-compatible with existing markdown bodies.

## Root cause

Two-layer mismatch between local-dev and cross-origin prod topology:

1. **Relative URL ambiguity** — Markdown bodies store the relative path `/api/v1/uploads/:id`. In local dev, Vite proxy maps `localhost:5173/api/*` → `localhost:3001` (same origin from browser perspective). In prod, the FE is on `*.netlify.app` and BE on `*.fly.dev` (different origins). The browser resolves the relative path against the FE origin → Netlify SPA fallback returns `index.html` for any unknown route → browser tries to render HTML as image → broken icon.

2. **Cross-origin `<img>` credentials are unreliable** — Even if we rewrite to absolute `https://onboarding-api.fly.dev/api/v1/uploads/:id`, the `GET` route currently uses `requireAuth` (line 175). Plain `<img>` cross-origin requests rely on third-party cookies, which Chrome is phasing out and Safari already blocks by default → request lands without `sid` cookie → 401.

## Fix approach

**Two-part fix**, pilot-scope appropriate:

1. **FE rewrite at render time**: In `apps/web/src/lib/markdown.ts` `afterSanitizeAttributes` hook, when an `<img>` src starts with `/api/v1/uploads/` or `/uploads/`, prefix it with `VITE_API_BASE_URL`'s origin (re-using the env var already used by `apps/web/src/lib/api.ts:8`). Existing markdown bodies in DB stay relative — render-time rewrite gives backfill for free, no migration needed.

2. **BE drop `requireAuth` on `GET /uploads/:id`**: UUIDv4 (~122 bits entropy) acts as the unguessable token. Matches pilot v1 access model in [FR-PROJ-001](../02-requirements.md#fr-proj-001--project-crud-minimal) ("all authenticated users see all data"), the standard pattern for user-content URLs (Slack, GitHub avatars, Google Drive view-link). The MIME whitelist + path-traversal guard on lines 156-163 stay; only the session check is removed. Update FR-UPLOAD-001 acceptance hints to document this trade-off explicitly.

**Why not Netlify proxy alternative**: adding `[[redirects]] /api/v1/*` → Fly keeps URLs relative but the cookie is scoped to `*.fly.dev` and never travels to Netlify edge — server-side proxy can't forward a cookie the browser never sent. The proxy alone doesn't fix auth, and combining proxy + drop-auth is strictly more moving parts than fix #2 above.

**Why not keep auth + crossorigin="use-credentials"**: would require sanitizer to inject the attribute, DOMPurify allowlist change, CORS preflight verification on uploads route, and still depends on `SameSite=None; Secure` third-party cookies which Chrome will block by 2026 per its phase-out roadmap.

- **Files to change**:
  - `apps/api/src/routes/uploads.ts` — drop `requireAuth` from `router.get("/:id", ...)` (line 175).
  - `apps/api/src/routes/uploads.ts` (optional defense in depth) — POST handler embeds absolute URL via `req.protocol + req.get("host") + path`, or simply leave relative and rely on FE rewrite.
  - `apps/web/src/lib/markdown.ts` — `afterSanitizeAttributes` `IMG` branch (line 90-95) prefixes relative upload paths with API origin.
  - `.specs/02-requirements.md` — FR-UPLOAD-001 acceptance hints amend (no longer requires auth on read).
- **Migration needed**: no.
